# Requirements Document

## Introduction

This document specifies the requirements for a PostgreSQL relational database schema supporting a blockchain-secured voting system. The schema replaces the existing SQLite database and introduces a fully normalized, auditable, privacy-preserving data model. Raw vote choices are never stored in the database; instead, each vote record links to an immutable on-chain transaction via a blockchain transaction hash. The schema supports voter lifecycle management, multi-election support with constituencies, candidate management, double-vote prevention via cryptographic nullifiers, and a comprehensive audit log.

## Glossary

- **Schema**: The PostgreSQL DDL definitions for all tables, constraints, and indexes.
- **Voter**: A registered individual eligible to cast a vote in an election.
- **Election**: A bounded voting event with a defined start date, end date, and status lifecycle.
- **Candidate**: An individual standing for election in a specific election.
- **Constituency**: A geographic or organizational grouping that scopes an election.
- **Vote**: A database record capturing the metadata of a single cast vote; no plaintext choice is stored.
- **Nullifier Hash**: A cryptographic commitment unique to one voter-election pair, used to prevent double voting.
- **Blockchain TX Hash**: The transaction hash of the on-chain record that permanently commits the vote hash to the blockchain.
- **Audit Log**: An append-only table recording all system actions for compliance and forensic investigation.
- **Fingerprint Hash**: A one-way hash of the voter's biometric or identity data used for privacy-preserving deduplication.
- **DDL**: Data Definition Language — SQL statements that define database structure (CREATE TABLE, CREATE INDEX, etc.).
- **DML**: Data Manipulation Language — SQL statements that insert or modify data (INSERT, UPDATE).
- **Migration**: A versioned, ordered SQL script that transforms the database schema from one state to another.

## Requirements

### Requirement 1: Voter Table

**User Story:** As a system administrator, I want a voters table with UUID primary keys, a status state machine, and a hashed identity fingerprint, so that voter eligibility can be tracked without exposing personally identifiable information.

#### Acceptance Criteria

1. THE Schema SHALL define a `voters` table with a UUID primary key column named `id` generated by default using `gen_random_uuid()`.
2. THE Schema SHALL define a `voters.status` column constrained to the enumerated values `REGISTERED`, `VOTED`, and `REVOKED`.
3. THE Schema SHALL define a `voters.fingerprint_hash` column of type `TEXT NOT NULL` with a non-unique index to support fast identity lookups.
4. THE Schema SHALL define `voters.name` as `TEXT NOT NULL`, `voters.email` as `TEXT NOT NULL UNIQUE`, `voters.created_at` as `TIMESTAMPTZ NOT NULL DEFAULT NOW()`, and `voters.updated_at` as `TIMESTAMPTZ NOT NULL DEFAULT NOW()`.
5. WHEN a voter record is inserted into the `voters` table, THE Schema SHALL default `voters.status` to `REGISTERED`.

---

### Requirement 2: Elections Table

**User Story:** As a system administrator, I want an elections table that tracks election lifecycle and constituency scoping, so that multiple elections can be managed independently.

#### Acceptance Criteria

1. THE Schema SHALL define an `elections` table with a UUID primary key column named `id` generated by default using `gen_random_uuid()`.
2. THE Schema SHALL define `elections.name` as `TEXT NOT NULL`, `elections.description` as `TEXT`, `elections.start_date` as `TIMESTAMPTZ NOT NULL`, `elections.end_date` as `TIMESTAMPTZ NOT NULL`.
3. THE Schema SHALL define an `elections.status` column constrained to the enumerated values `UPCOMING`, `ACTIVE`, and `CLOSED`.
4. THE Schema SHALL define `elections.constituency_id` as a UUID foreign key referencing the `constituencies` table with `ON DELETE RESTRICT`.
5. WHEN an election record is inserted into the `elections` table, THE Schema SHALL default `elections.status` to `UPCOMING`.
6. IF `elections.end_date` is less than or equal to `elections.start_date`, THEN THE Schema SHALL reject the record via a CHECK constraint.

---

### Requirement 3: Constituencies Table

**User Story:** As a system administrator, I want a constituencies table to group elections by geographic or organizational unit, so that elections can be scoped to the correct eligible population.

#### Acceptance Criteria

1. THE Schema SHALL define a `constituencies` table with a UUID primary key column named `id` generated by default using `gen_random_uuid()`.
2. THE Schema SHALL define `constituencies.name` as `TEXT NOT NULL UNIQUE` and `constituencies.description` as `TEXT`.

---

### Requirement 4: Candidates Table

**User Story:** As an election administrator, I want a candidates table linked to elections, so that each candidate's details and active status can be managed per election.

#### Acceptance Criteria

1. THE Schema SHALL define a `candidates` table with a UUID primary key column named `id` generated by default using `gen_random_uuid()`.
2. THE Schema SHALL define `candidates.name` as `TEXT NOT NULL`, `candidates.party` as `TEXT NOT NULL`, and `candidates.position` as `TEXT NOT NULL`.
3. THE Schema SHALL define `candidates.election_id` as a UUID foreign key referencing the `elections` table with `ON DELETE RESTRICT`.
4. THE Schema SHALL define `candidates.active` as `BOOLEAN NOT NULL DEFAULT TRUE`.

---

### Requirement 5: Votes Table

**User Story:** As a system operator, I want a votes table that stores only vote metadata linked to an immutable blockchain transaction, so that the system is auditable without exposing individual vote choices.

#### Acceptance Criteria

1. THE Schema SHALL define a `votes` table with a UUID primary key column named `id` generated by default using `gen_random_uuid()`.
2. THE Schema SHALL define `votes.voter_id` as a UUID foreign key referencing `voters.id` with `ON DELETE RESTRICT`.
3. THE Schema SHALL define `votes.candidate_id` as a UUID foreign key referencing `candidates.id` with `ON DELETE RESTRICT`.
4. THE Schema SHALL define `votes.election_id` as a UUID foreign key referencing `elections.id` with `ON DELETE RESTRICT`.
5. THE Schema SHALL define `votes.nullifier_hash` as `TEXT NOT NULL` with a `UNIQUE` constraint to prevent any voter from casting more than one vote per election.
6. THE Schema SHALL define `votes.blockchain_tx_hash` as `TEXT NOT NULL` to store the on-chain transaction reference.
7. THE Schema SHALL define `votes.ip_address_hash` as `TEXT` to store the hashed IP address of the submitting client.
8. THE Schema SHALL define `votes.submitted_at` as `TIMESTAMPTZ NOT NULL DEFAULT NOW()`.
9. THE Schema SHALL NOT define any column in the `votes` table that stores a plaintext vote choice or candidate preference.

---

### Requirement 6: Audit Log Table

**User Story:** As a compliance officer, I want an audit_log table that records every system action with structured metadata, so that security events can be investigated and the system can demonstrate accountability.

#### Acceptance Criteria

1. THE Schema SHALL define an `audit_log` table with a UUID primary key column named `id` generated by default using `gen_random_uuid()`.
2. THE Schema SHALL define `audit_log.action` as `TEXT NOT NULL` to capture the action type (e.g., `LOGIN_ATTEMPT`, `VOTE_SUBMITTED`, `BLOCKCHAIN_COMMIT`).
3. THE Schema SHALL define `audit_log.actor_id` as `TEXT` to reference the voter or admin performing the action (nullable to support unauthenticated events).
4. THE Schema SHALL define `audit_log.ip_address` as `TEXT` to record the originating IP address.
5. THE Schema SHALL define `audit_log.timestamp` as `TIMESTAMPTZ NOT NULL DEFAULT NOW()` with an index to support fast time-range queries.
6. THE Schema SHALL define `audit_log.details` as `JSONB` to store flexible structured metadata about the event.
7. THE Schema SHALL define `audit_log.success` as `BOOLEAN NOT NULL` to record whether the action completed successfully.

---

### Requirement 7: Indexes and Constraints

**User Story:** As a database administrator, I want indexes and referential integrity constraints on all tables, so that queries are performant and data consistency is enforced at the database level.

#### Acceptance Criteria

1. THE Schema SHALL define a unique index on `votes.nullifier_hash` to enforce the double-vote prevention invariant at the database level.
2. THE Schema SHALL define an index on `voters.fingerprint_hash` to support fast identity deduplication lookups.
3. THE Schema SHALL define an index on `audit_log.timestamp` to support fast forensic time-range queries.
4. THE Schema SHALL define an index on `votes.election_id` to support fast tally queries scoped to a single election.
5. THE Schema SHALL define an index on `candidates.election_id` to support fast candidate lookups per election.
6. THE Schema SHALL define all foreign key columns with `ON DELETE RESTRICT` to prevent orphaned records.
7. THE Schema SHALL apply UUID primary keys to all tables to prevent row enumeration attacks.

---

### Requirement 8: Seed Data

**User Story:** As a developer, I want a seed script with representative test data, so that the schema can be validated and the API can be developed and tested against realistic data.

#### Acceptance Criteria

1. THE Seed Script SHALL insert exactly 3 constituency records into the `constituencies` table.
2. THE Seed Script SHALL insert exactly 1 election record with status `ACTIVE` referencing one of the seeded constituencies.
3. THE Seed Script SHALL insert exactly 5 voter records with status `REGISTERED` into the `voters` table.
4. THE Seed Script SHALL insert exactly 3 candidate records referencing the seeded election into the `candidates` table.
5. THE Seed Script SHALL insert at least 3 audit log entries representing distinct action types (e.g., `LOGIN_ATTEMPT`, `VOTE_SUBMITTED`, `BLOCKCHAIN_COMMIT`).
6. THE Seed Script SHALL use `INSERT ... ON CONFLICT DO NOTHING` to be safely re-runnable without producing duplicate data errors.

---

### Requirement 9: Migration Strategy

**User Story:** As a developer, I want a versioned migration approach, so that the schema can be applied to a fresh database and evolved in future iterations without data loss.

#### Acceptance Criteria

1. THE Schema DDL SHALL be contained in a single `schema.sql` file that can be applied to a fresh PostgreSQL database using `psql -f schema.sql`.
2. THE Schema DDL SHALL use `CREATE TABLE IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS` to be idempotent on re-execution against an existing database.
3. THE Schema DDL SHALL enable the `pgcrypto` extension (or use `gen_random_uuid()` from `pg_catalog`) to support UUID generation.
4. THE Seed Script SHALL be contained in a separate `seed.sql` file that can be applied independently after the schema is created.

---

### Requirement 10: Documentation

**User Story:** As a new developer on the project, I want a README that explains how to set up the database, so that I can get a local environment running without tribal knowledge.

#### Acceptance Criteria

1. THE README SHALL document the PostgreSQL version requirement (14 or higher).
2. THE README SHALL provide the exact `psql` commands required to create the database, apply the schema, and run the seed script.
3. THE README SHALL list all tables with a one-line description of each table's purpose.
4. THE README SHALL document the required environment variables for connecting the existing Node.js API to the PostgreSQL database.
